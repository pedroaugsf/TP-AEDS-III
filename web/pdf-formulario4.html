1. Taxa de Compressão — Algoritmo de Huffman

Método: Codificação por prefixos de comprimento variável (árvore de Huffman). A tabela de frequências de cada byte é gravada no cabeçalho do fluxo comprimido para reconstrução determinística da árvore na descompressão.

(pacote TAR com os 22 arquivos de ./dados/)		 8.874 bytes (~8,7 KB)
b) Tamanho do arquivo comprimido
(./backups/nutritrack_huffman.huff)		 5.073 bytes (~5,0 KB)
c) Cálculo da taxa de compressão taxa = (1 − comprimido / original) × 100
taxa = (1 − 5073 / 8874) × 100
taxa ≈ 42,83 %
Razão original:comprimido = 1,75 : 1
d) Interpretação O Huffman atribui códigos de tamanho variável inversamente proporcionais à frequência de cada byte. Como os arquivos .db contêm muitos bytes de preenchimento (lápide 0x20, espaços livres, campos fixos e zeros de padding), a distribuição de frequências é muito enviesada, favorecendo o algoritmo. Ele atinge a entropia de ordem 0 (símbolo a símbolo), mas não captura redundâncias em sequências repetidas — razão pela qual o LZW superou-o neste caso.

2. Taxa de Compressão — Algoritmo de LZW

Método: Dicionário adaptativo (Lempel–Ziv–Welch) com códigos de 16 bits. O dicionário começa com as 256 entradas de byte único e cresce dinamicamente até 65.536 entradas; compressor e descompressor param exatamente no mesmo ponto, garantindo sincronismo.
ItemValor
a) Tamanho do arquivo original
(mesmo pacote TAR com os 22 arquivos de ./dados/)		 8.874 bytes (~8,7 KB)
b) Tamanho do arquivo comprimido
(./backups/nutritrack_lzw.lzw)		 4.269 bytes (~4,2 KB)
c) Cálculo da taxa de compressão taxa = (1 − comprimido / original) × 100
taxa = (1 − 4269 / 8874) × 100
taxa ≈ 51,89 %
Razão original:comprimido = 2,08 : 1
d) Interpretação O LZW constrói um dicionário de sequências recorrentes durante a leitura dos dados. Os arquivos binários .db possuem sequências estruturais repetidas: cabeçalhos fixos, blocos de bytes nulos (campos long zerados), strings UTF-8 comuns e padrões de índice. O LZW substitui essas sequências inteiras por um único código de 16 bits, capturando redundâncias que o Huffman ignora por tratar cada byte de forma independente. Por isso, o LZW alcançou taxa 9,1 pontos percentuais superior ao Huffman neste conjunto de dados.

3. Dificuldades ao Implementar Huffman e LZW — e Como Foram Resolvidas

3.1 Gerar um único arquivo compactado com múltiplos arquivos de dados

Os dados do sistema estão distribuídos em várias pastas e arquivos (usuario.db, alimento.db, índices .idx.*.db, B+ .idx.bmais.db etc.). O requisito exige um único arquivo compactado.

Solução: foi implementado o empacotador app.compression.Backup, que percorre recursivamente ./dados/ e serializa cada arquivo no formato: 

[assinatura: 6 bytes "NTBKP1"]
[quantidade: int]
para cada arquivo:
  [tamanho do nome: short] [nome relativo: UTF-8]
  [tamanho do conteúdo: long] [conteúdo: byte[]]

O pacote resultante — um único byte[] — é então comprimido inteiramente por Huffman ou LZW, gerando o arquivo final. A restauração reconstrói a árvore de diretórios com proteção contra path traversal.

3.2 Huffman — persistir a árvore de decodificação

Para descomprimir é necessário reconstruir exatamente a mesma árvore de Huffman usada na compressão.

Solução: em vez de serializar a árvore (estrutura complexa), o cabeçalho do fluxo grava a tabela de frequências (pares [byte símbolo, long frequência]). Na descompressão, a fila de prioridade reconstrói a árvore de forma determinística — qualquer entrada com a mesma tabela de frequências produz a mesma árvore.

3.3 Huffman — empacotamento de bits (bit-packing)

Os códigos de Huffman têm comprimentos variáveis (1 a vários bits) e não se alinham a fronteiras de byte. Gravar um código por byte desperdiçaria todo o ganho.

Solução: uso de um bit buffer inteiro que acumula bits e descarrega 1 byte sempre que 8 bits são preenchidos. O último byte é completado com zeros à direita (padding). Na descompressão, o número exato de bytes originais é registrado no cabeçalho (tamanhoOriginal: int), impedindo a leitura dos bits de padding.

3.4 Huffman — arquivo com um único símbolo distinto

Se todos os bytes do arquivo são iguais, a árvore de Huffman tem apenas um nó (a raiz é folha). A geração recursiva de códigos não consegue descer por nenhum ramo e o prefixo fica vazio.

Solução: caso o prefixo seja vazio ao chegar em uma folha, o código atribuído é "0" (um único bit). Na descompressão, quando a raiz já é folha, os bytes de saída são preenchidos diretamente com o símbolo da raiz sem percorrer a árvore.

3.5 LZW — sincronizar o dicionário entre compressor e descompressor

Compressor e descompressor precisam construir dicionários idênticos passo a passo, sem transmitir o dicionário explicitamente. O caso especial KwKwK ocorre quando o descompressor recebe um código que ainda não existe em seu dicionário (o compressor o criou no passo anterior, mas o descompressor ainda não processou a entrada correspondente).

Solução: ao receber um código igual ao próximo a ser inserido no dicionário, o descompressor infere que a nova sequência é a sequência anterior mais seu próprio primeiro byte. O dicionário para de crescer exatamente nos mesmos 65.536 entradas nos dois lados, garantindo sincronismo total sem nenhuma informação extra no fluxo.

3.6 Verificação de integridade

Erros na implementação (ex.: off-by-one no bit buffer, código errado no caso KwKwK) são silenciosos — o arquivo é gerado sem erro, mas os dados descomprimidos diferem do original.

Solução: o CompressaoService executa uma verificação round-trip automática antes de gravar o arquivo de backup: comprime o pacote, descomprime imediatamente e compara o resultado byte a byte com o original via Arrays.equals(). Se a verificação falhar, o arquivo não é gerado e o campo integridadeOk retorna false. Nos testes realizados, ambos os algoritmos passaram em 100% dos casos, incluindo entradas vazias, mono-símbolo e dados aleatórios.
