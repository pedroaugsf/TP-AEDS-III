Resumo. Este documento descreve o NutriTrack, um sistema para cadastro e gerenciamento de usuários, alimentos, refeições e registro de consumo nutricional. O sistema implementa operações CRUD completas com persistência em arquivos binários, utilizando cabeçalho com controle de ID sequencial e exclusão lógica por lápide, além de uma lista encadeada para reaproveitamento de espaço. Na Fase 2 foram acrescentados índices em Hash Extensível (índice primário e índice 1:N) e uma API REST com front-end SPA. A arquitetura segue o padrão MVC + DAO em Java.

1. Introdução

O acompanhamento nutricional é uma necessidade crescente. O NutriTrack foi desenvolvido para permitir que um usuário registre suas refeições e os alimentos consumidos, com informações detalhadas de macronutrientes (calorias, proteínas, carboidratos e gorduras).

O sistema foi construído em Java, seguindo o padrão MVC + DAO, e armazena todos os dados exclusivamente em arquivos binários — sem uso de banco de dados relacional. Cada arquivo possui cabeçalho com controle de ID auto-incremental e ponteiro para uma lista encadeada de registros removidos (lápide), permitindo o reaproveitamento de espaço em disco.

2. Descrição do Problema

O sistema deve permitir o cadastro e gerenciamento de Usuários, Alimentos, Refeições e Consumos, possibilitando registrar quais alimentos foram consumidos em cada refeição e em qual quantidade (em gramas). Os dados devem ser persistidos em arquivo binário com cabeçalho e controle de exclusão lógica por lápide.

3. Objetivo do Trabalho

  • Desenvolver um sistema CRUD completo para as 4 entidades do domínio nutricional.
  • Garantir persistência em arquivos binários (.db) com cabeçalho, lápide e reaproveitamento de espaço.
  • Implementar relacionamentos 1:N (Usuário → Refeição) e N:N (Refeição ↔ Alimento via Consumo).
  • Validar chaves estrangeiras (FK) na camada Controller.
  • Acelerar buscas com Hash Extensível (índice primário PK e índice 1:N FK→PK).
  • Fornecer documentação com DCU, DER e Arquitetura.

8.1 Descrição das Entidades

Usuário: Representa a pessoa que utiliza o sistema. Possui atributos simples (nome, email), um atributo do tipo data (dataNascimento, armazenado como epochDay) e um atributo multivalorado (telefones).

Alimento: Representa um item alimentar com informações nutricionais por 100g. Contém atributos float para macronutrientes e um atributo multivalorado (tags).

Refeição: Representa uma refeição realizada por um usuário em determinada data. Possui chave estrangeira usuarioId (1:N com Usuário) e campos tipo e observação.

Consumo: Entidade associativa que implementa o relacionamento N:N entre Refeição e Alimento, com o atributo quantidadeGramas.

8.2 Relacionamentos

RelacionamentoCardinalidadeDescrição
Usuário → Refeição1 : NUm usuário realiza várias refeições.
Refeição → Consumo1 : NUma refeição contém vários itens consumidos.
Alimento → Consumo1 : NUm alimento pode ser consumido em várias refeições.

10.2 Registro

CampoTipoTamanhoDescrição

Dadosbyte[]variávelObjeto serializado via toByteArray().

10.3 Reaproveitamento de Espaço

Quando um registro é removido, a lápide é marcada como '*' e a posição é inserida em uma lista encadeada ordenada por tamanho (best-fit). No create/update que necessita de mais espaço, o sistema busca na lista um espaço removido que caiba os novos dados. Caso não exista, o registro é gravado no final do arquivo.

12. Hash Extensível (Fase 2)

12.1 Visão geral

Foram adicionados dois tipos de índice baseados em Hash Extensível:

  1. Índice primário (uma por tabela): mapeia id (PK) → endereço no .db. read, update e delete passam a fazer um único seek em O(1) médio.
  2. Índice 1:N: mapeia refeicaoId (FK) → consumoId (PK). Permite recuperar todos os consumos de uma refeição em uma única consulta hash. Também viabiliza a remoção em cascata.

12.2 Estrutura física

ArquivoConteúdo
{nome}.idx.d.dbDiretório: int profundidadeGlobal + 2^profGlobal ponteiros long.
{nome}.idx.c.dbCestos: byte profundidadeLocal + short qtd + slots de tamanho fixo.
Tipo de entradaCamposTamanho fixo
ParIDEndereco (primário)int id + long endereco12 bytes
ParIDID (1:N)int chave1 (FK) + int chave2 (PK)8 bytes

12.3 Operações

  • Inserção: idx = hashCode & (2^profGlobal − 1). Se o cesto enche, ocorre split com possível duplicação do diretório.
  • Leitura por chave: 2 seeks (diretório + cesto). read(K) retorna todos os elementos com a chave (essencial para 1:N).
  • Remoção: identifica o slot pelo equals(). No 1:N considera o par (FK, PK) para não remover entradas erradas.

12.5 Reconstrução automática

Tanto o índice primário quanto o 1:N possuem rotina de reconstrução: se o arquivo de índice está vazio mas o .db de dados já contém registros, o índice é reconstruído na inicialização varrendo o arquivo de dados.

13. Front-end e API REST

O front-end é uma SPA estática em web/ servida pelo servidor HTTP embutido em app.Servidor (porta 8080, sem dependências externas). A interface oferece CRUD completo das 4 entidades e demonstra o índice 1:N.

17. Considerações Finais

O NutriTrack atende plenamente aos requisitos das duas fases. A Fase 1 entregou o CRUD completo com persistência binária, lápide e reaproveitamento de espaço. A Fase 2 acrescentou o Hash Extensível como índice primário (acelerando todas as operações por chave) e como índice 1:N (acelerando consultas de relacionamento e habilitando cascade), além de uma camada de transporte HTTP/REST e um front-end SPA. A camada de validação garante a integridade referencial entre as entidades.
