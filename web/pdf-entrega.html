Parte 1 — Instruções de Compilação e Execução

Pré-requisitos

  • JDK 17+ (testado com OpenJDK 21) instalado e no PATH
  • Sem dependências externas — todo o servidor HTTP é embutido via com.sun.net.httpserver

Passo a Passo — Windows (PowerShell)

# 1. Entrar na pasta do projeto
Set-Location 'caminho\para\TP-AEDS-III'

# 2. Compilar (gera ./out)
if (Test-Path out) { Remove-Item -Recurse -Force out }
New-Item -ItemType Directory out | Out-Null
javac -d out (Get-ChildItem -Recurse src -Filter *.java | ForEach-Object { $_.FullName })

# 3a. Executar — Servidor HTTP + front-end (RECOMENDADO)
java -cp out app.Servidor
#    → acessar no navegador: http://localhost:8080

# 3b. Executar — CRUD via menu de console
java -cp out app.ConsoleApp

# 3c. Executar — Demo guiado (cria dados de exemplo)
java -cp out app.Main

Passo a Passo — Linux / macOS

mkdir -p out
javac -d out $(find src -name "*.java")
java -cp out app.Servidor   # acesse http://localhost:8080

Gerar Backup de Compressão (Fase IV)

Com o servidor rodando, acesse a aba "Backup & Compressão" no menu lateral e clique em Gerar backup Huffman / Gerar backup LZW. Os arquivos são gravados em ./backups/.

Via PowerShell (com servidor rodando em porta 8080):

# Gerar backups
Invoke-RestMethod -Method Post http://localhost:8080/api/backup/huffman
Invoke-RestMethod -Method Post http://localhost:8080/api/backup/lzw

# Baixar o arquivo compactado
Invoke-RestMethod http://localhost:8080/api/backup/download/huffman -OutFile nutritrack.huff
Invoke-RestMethod http://localhost:8080/api/backup/download/lzw     -OutFile nutritrack.lzw

# Restaurar em ./dados_restaurado/
Invoke-RestMethod -Method Post http://localhost:8080/api/backup/restaurar/huffman

Notas Importantes

  • A pasta ./dados/ é criada automaticamente na primeira execução.
  • Apagar ./dados/ reinicia o banco. Apagar apenas os *.idx.*.db reconstrói somente os índices.
  • Para popular dados de demonstração: .\seed.ps1 (com servidor rodando).
  • Porta padrão: 8080. Pode ser alterada via variável de ambiente: $env:PORT = "9090".
Parte 2 — Documentação Técnica (Fases 1 a 4)

1. Introdução

O NutriTrack é um sistema para cadastro e gerenciamento de usuários, alimentos, refeições e registros de consumo nutricional. Construído em Java puro (sem frameworks ou banco de dados), persiste todos os dados em arquivos binários com estrutura própria de cabeçalho, lápide e reaproveitamento de espaço.

O trabalho foi dividido em quatro fases: a Fase 1 implementou o CRUD completo; a Fase 2 adicionou índices em Hash Extensível; a Fase 3 introduziu Árvore B+ e relacionamento N:N; a Fase 4 implementou compressão Huffman e LZW com geração de backup único.

5. Fase 2 — Hash Extensível

Dois tipos de índice foram implementados:

  • Índice primário (ParIDEndereco): mapeia id → endereço no .db para cada tabela. Operações de leitura/atualização/remoção passam a fazer O(1) médio.
  • Índice 1:N (ParIDID): mapeia refeicaoId → consumoId. Permite listar todos os consumos de uma refeição e viabiliza o cascade delete.
ArquivoConteúdo
{nome}.idx.d.dbDiretório: profundidade global + 2^prof ponteiros long
{nome}.idx.c.dbCestos: profundidade local + quantidade + slots de tamanho fixo

Os índices são reconstruídos automaticamente na inicialização se os arquivos .idx.*.db estiverem ausentes.

6. Fase 3 — Árvore B+ e Relacionamento N:N

6.1 Árvore B+ Genérica

Implementada do zero em app.dao.ArvoreBMais<T> com folhas encadeadas, ordem 8 e página de 595 bytes. Indexa nome → id do Alimento em ./dados/alimento/alimento.idx.bmais.db.

A consulta GET /api/alimento/ordenado percorre a lista encadeada de folhas em ordem crescente de chave — sem ordenação em memória.

6.2 Relacionamento N:N — Favoritos

Tabela intermediária Favorito com unicidade do par (usuarioId, alimentoId) verificada antes de cada inserção. Dois índices Hash bidirecionais:

ÍndiceChaveConsulta atendida
favorito_por_usuariousuarioIdtodos os alimentos favoritados por um usuário
favorito_por_alimentoalimentoIdtodos os usuários que favoritaram um alimento

Cascade: remover Usuário ou Alimento remove os Favoritos vinculados e limpa ambos os índices.

7. Fase 4 — Compressão de Dados (Huffman e LZW)

7.1 Arquitetura de Backup

  1. Empacotamento (TAR): app.compression.Backup percorre recursivamente ./dados/ e serializa cada arquivo com seu caminho relativo, tamanho e conteúdo num único byte[].
  2. Compressão: o pacote é comprimido por Huffman ou LZW, gerando um único arquivo em ./backups/. O cabeçalho do arquivo final identifica o algoritmo e o tamanho original.
  3. Verificação round-trip: antes de gravar, o serviço comprime → descomprime → compara byte a byte via Arrays.equals().

7.2 Huffman

Codificação por prefixos de comprimento variável. A tabela de frequências dos 256 bytes possíveis é gravada no cabeçalho do fluxo, permitindo reconstrução determinística da árvore na descompressão. Casos tratados: arquivo vazio, símbolo único (código "0"), bit-packing com padding no último byte.

7.3 LZW

Dicionário adaptativo com códigos de 16 bits, iniciando com 256 entradas de byte único. Para de crescer em 65.536 entradas. Compressor e descompressor param no mesmo ponto, garantindo sincronismo. Caso especial KwKwK tratado explicitamente.

Parte 3 — Formulário Técnico da Fase IV

Formulário exigido pelo enunciado da Fase IV — Compressão de Dados.

Checklist de Requisitos Atendidos:
  • Compressão obrigatória sobre todos os arquivos de dados de ./dados/
  • Resultado em arquivo único compactado — funciona como backup completo
  • Compactação exclusivamente a nível de arquivo (sem bibliotecas externas de compressão)
  • Algoritmo Huffman implementado do zero em app.compression.Huffman
  • Algoritmo LZW implementado do zero em app.compression.LZW

Questão 1 — Qual foi a taxa de compressão obtida com o algoritmo de Huffman?
ItemValor
a) Tamanho do arquivo original
Pacote TAR com os 22 arquivos de ./dados/		 8.874 bytes (~8,7 KB)
b) Tamanho do arquivo comprimido
./backups/nutritrack_huffman.huff		 5.073 bytes (~5,0 KB)
c) Cálculo da taxa taxa = (1 − comprimido / original) × 100
taxa = (1 − 5073 / 8874) × 100
taxa ≈ 42,83 %
Razão original : comprimido = 1,75 : 1
d) Interpretação do resultado O Huffman atribui códigos de tamanho variável inversamente proporcionais à frequência de cada byte (símbolo mais frequente = código mais curto). Os arquivos .db do sistema possuem alta redundância estatística: bytes de preenchimento (0x20 = lápide ativa), zeros de padding em campos long e sequências de índice se repetem muito. Por isso a distribuição de frequências é enviesada, favorecendo o Huffman. Ele atinge a entropia de ordem 0, mas não enxerga redundância de sequências — razão pela qual o LZW superou-o neste conjunto de dados.

d) Interpretação do resultado O LZW constrói um dicionário de sequências recorrentes em tempo de execução. Questão 3 — Quais dificuldades surgiram ao implementar Huffman e LZW e como foram resolvidas?

3.1 — Gerar um único arquivo compactado a partir de múltiplos arquivos

Problema: os dados estão distribuídos em várias pastas e arquivos de ./dados/. O requisito exige um único arquivo compactado.

Solução: implementado o empacotador app.compression.Backup (estilo TAR), que percorre recursivamente ./dados/ e serializa cada arquivo no formato:

[assinatura "NTBKP1": 6 bytes]  [qtd arquivos: int]
para cada arquivo:
  [tamanho do nome: short]  [nome relativo UTF-8]  [tamanho do conteúdo: long]  [bytes do arquivo]

O pacote resultante é então comprimido inteiramente. A restauração reconstrói a árvore de diretórios com proteção contra path traversal (verifica que o destino final não sai da pasta raiz).

3.2 — Huffman: persistir a árvore de decodificação

Problema: para descomprimir é necessário reconstruir exatamente a mesma árvore de Huffman usada na compressão.

Solução: em vez de serializar a estrutura da árvore, o cabeçalho do fluxo grava a tabela de frequências (pares [byte símbolo, long frequência]). Na descompressão, a fila de prioridade reconstrói a árvore de forma determinística — mesma tabela de frequências → mesma árvore.

3.3 — Huffman: empacotamento de bits (bit-packing)

Problema: os códigos de Huffman têm comprimentos variáveis e não se alinham a fronteiras de byte. Gravar um código por byte eliminaria todo o ganho de compressão.

Solução: uso de um bit buffer inteiro que acumula bits e descarrega 1 byte sempre que 8 bits são preenchidos. O último byte é completado com zeros à direita (padding). O cabeçalho grava o tamanhoOriginal em bytes para impedir a leitura dos bits de padding na descompressão.

3.4 — Huffman: arquivo com um único byte distinto

Problema: quando todos os bytes da entrada são iguais, a árvore tem apenas um nó (raiz = folha). A geração recursiva de códigos não consegue descer por nenhum ramo e o prefixo fica vazio.

Solução: se o prefixo estiver vazio ao chegar em uma folha, o código atribuído é "0" (um único bit). Na descompressão, quando a raiz já é folha, os bytes de saída são preenchidos diretamente com o símbolo da raiz sem percorrer nenhum ramo.

3.5 — LZW: sincronizar o dicionário (caso especial KwKwK)

Problema: compressor e descompressor precisam construir dicionários idênticos sem transmiti-lo explicitamente. O caso especial KwKwK ocorre quando o descompressor recebe um código que ainda não existe no seu dicionário (o compressor o acabou de criar, mas o descompressor ainda não processou a entrada correspondente).

Solução: ao receber um código igual ao próximo a ser inserido no dicionário, o descompressor infere que a nova sequência é a sequência anterior mais seu próprio primeiro byte. Além disso, os dois lados param de crescer exatamente em 65.536 entradas, garantindo sincronismo total sem nenhum dado extra no fluxo.

3.6 — Verificação de integridade (round-trip)

Problema: erros sutis de implementação (ex.: off-by-one no bit buffer, caso KwKwK mal tratado) são silenciosos — o arquivo é gerado sem exceção, mas os dados descomprimidos divergem do original.

Solução: o CompressaoService executa uma verificação round-trip automática antes de gravar o arquivo de backup: comprime o pacote → descomprime imediatamente → compara byte a byte com Arrays.equals(). O campo integridadeOk é retornado pela API e exibido no front-end. Ambos os algoritmos passaram em 100% dos casos de teste (entrada vazia, mono-símbolo, dados aleatórios e dados reais).
