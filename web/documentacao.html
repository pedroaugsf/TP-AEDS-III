Resumo. Este documento descreve o NutriTrack, um sistema para cadastro e gerenciamento de usuários, alimentos, refeições e registro de consumo nutricional. O sistema implementa operações CRUD completas com persistência em arquivos binários, utilizando cabeçalho com controle de ID sequencial e exclusão lógica por lápide, além de uma lista encadeada para reaproveitamento de espaço. A arquitetura segue o padrão MVC + DAO em Java, com interface mínima em HTML/CSS.

1. Introdução

O acompanhamento nutricional é uma necessidade crescente. O NutriTrack foi desenvolvido para permitir que um usuário registre suas refeições e os alimentos consumidos, com informações detalhadas de macronutrientes (calorias, proteínas, carboidratos e gorduras).

O sistema foi construído em Java, seguindo o padrão MVC + DAO, e armazena todos os dados exclusivamente em arquivos binários — sem uso de banco de dados relacional. Cada arquivo possui cabeçalho com controle de ID auto-incremental e ponteiro para uma lista encadeada de registros removidos (lápide), permitindo o reaproveitamento de espaço em disco.

2. Descrição do Problema

O sistema deve permitir o cadastro e gerenciamento de Usuários, Alimentos, Refeições e Consumos, possibilitando registrar quais alimentos foram consumidos em cada refeição e em qual quantidade (em gramas). Os dados devem ser persistidos em arquivo binário com cabeçalho e controle de exclusão lógica por lápide.

3. Objetivo do Trabalho

  • Desenvolver um sistema CRUD completo para as 4 entidades do domínio nutricional.
  • Garantir persistência em arquivos binários (.db) com cabeçalho, lápide e reaproveitamento de espaço.
  • Implementar relacionamentos 1:N (Usuário → Refeição) e N:N (Refeição ↔ Alimento via Consumo).
  • Validar chaves estrangeiras (FK) na camada Controller.
  • Fornecer documentação com DCU, DER e Arquitetura.

8.1 Descrição das Entidades

Usuário: Representa a pessoa que utiliza o sistema. Possui atributos simples (nome, email), um atributo do tipo data (dataNascimento, armazenado como epochDay) e um atributo multivalorado (telefones, armazenado como lista de strings).

Alimento: Representa um item alimentar com informações nutricionais por 100g. Contém atributos float para macronutrientes e um atributo multivalorado (tags).

Refeição: Representa uma refeição realizada por um usuário em determinada data. Possui chave estrangeira usuarioId (1:N com Usuário) e campos tipo e observação.

Consumo: Entidade associativa (tabela intermediária) que implementa o relacionamento N:N entre Refeição e Alimento, com o atributo quantidadeGramas.

8.2 Relacionamentos

Refeição → Consumo1 : NUma refeição contém vários itens consumidos.

10. Estrutura do Arquivo Binário

Cada arquivo .db segue a estrutura abaixo:

10.1 Cabeçalho (12 bytes)

OffsetTipoTamanhoDescrição
0int4 bytesÚltimo ID atribuído (auto-incremento).
4long8 bytesPonteiro (posição) da cabeça da lista de registros removidos. Valor -1 indica lista vazia.

10.2 Registro

CampoTipoTamanhoDescrição
Lápidebyte1 byte' ' (0x20) = ativo; '*' (0x2A) = removido.
Tamanhoshort2 bytesTamanho do array de dados em bytes.
Dadosbyte[]variávelObjeto serializado via toByteArray().

10.3 Reaproveitamento de Espaço

Quando um registro é removido, a lápide é marcada como '*' e a posição é inserida em uma lista encadeada ordenada por tamanho (best-fit). No momento de um create ou update que necessita de mais espaço, o sistema busca na lista um espaço removido que caiba os novos dados. Caso não exista, o registro é gravado no final do arquivo.

14. Considerações Finais

O NutriTrack atende aos requisitos propostos: implementa CRUD completo para 4 entidades inter-relacionadas, com persistência exclusivamente em arquivos binários. A classe genérica Arquivo<T> centraliza toda a lógica de acesso a disco — cabeçalho com ID auto-incremental, lápide para exclusão lógica e lista encadeada ordenada para reaproveitamento de espaço. A validação de chaves estrangeiras é feita na camada Controller, garantindo integridade referencial entre as entidades.

15. Fase 2 — Índices em Hash Extensível

15.1 Visão geral

Na Fase 2 o projeto recebeu duas adições estruturais que mudam a forma como os registros são localizados em disco:

  1. Índice primário (uma por tabela) baseado em Hash Extensível, mapeando id (PK) → endereço no .db. Com isso, read, update e delete deixaram de varrer linearmente o arquivo de dados e passaram a fazer um único seek em O(1) médio.
  2. Índice 1:N (uma instância) baseado em Hash Extensível, mapeando refeicaoId (FK) → consumoId (PK). Permite recuperar todos os consumos de uma refeição em uma única consulta hash, sem varrer o arquivo de consumos. Esse índice também viabiliza a remoção em cascata.

15.2 Estrutura física do Hash Extensível

Cada índice é composto por dois arquivos:

ArquivoConteúdo
{nome}.idx.d.db Diretório: int profundidadeGlobal seguido de 2^profGlobal ponteiros long (endereços de cestos).
{nome}.idx.c.db Cestos (buckets): sequência de cestos de tamanho fixo. Cada cesto contém byte profundidadeLocal + short quantidade + vetor de slots (cada slot tem o tamanho fixo da entrada).

Tamanho de uma entrada:

Tipo de entradaCamposTamanho fixo
ParIDEndereco (índice primário) int id + long endereco 12 bytes
ParIDID (índice 1:N) int chave1 (FK) + int chave2 (PK) 8 bytes

15.3 Operações

  • Inserção: calcula idx = hashCode & (2^profGlobal − 1), lê o ponteiro do diretório e tenta inserir o slot no cesto correspondente. Se o cesto está cheio, ocorre split: a profundidade local é incrementada, um novo cesto é alocado no fim do arquivo de cestos, os ponteiros do diretório são reescritos e os elementos são redistribuídos. Caso a profundidade local ultrapasse a global, o diretório é dobrado (cada ponteiro original passa a apontar para duas entradas).
  • Leitura por chave: um seek no diretório, um seek no cesto e varredura sequencial dentro do cesto (≤ slots por cesto). O método read(int) retorna todos os elementos cujo hashCode() bate, o que é essencial para o índice 1:N onde várias entradas compartilham a mesma chave.
  • Remoção: identifica o slot pelo equals() (no índice primário basta a chave; no índice 1:N considera o par (FK, PK) para evitar remover entradas erradas).

A integridade referencial é garantida em duas frentes: (i) os controllers validam, antes de inserir/atualizar, que as FKs apontam para registros existentes (usando exists(id), que reaproveita o índice primário); (ii) ao remover uma refeição, o RefeicaoController consulta o índice 1:N e dispara o cascade chamando ConsumoDAO.removerPorRefeicao(id), que apaga cada consumo associado e remove a respectiva entrada do índice 1:N.

15.5 Reconstrução automática

Tanto o índice primário quanto o 1:N possuem rotina de reconstrução: se o arquivo de índice está vazio mas o .db de dados já contém registros, o índice é reconstruído na inicialização varrendo o arquivo de dados. Isso protege contra a perda dos arquivos de índice e permite remover apenas a pasta de índices para forçar a reconstrução.

15.6 Persistência e sincronização

  • Tudo é gravado via RandomAccessFile; cada operação faz seek + write imediatamente, sem cache em memória.
  • Os índices são atualizados na mesma operação que altera o arquivo de dados (inserção, atualização que muda endereço, remoção).
  • O close() é chamado por um shutdown hook no servidor, garantindo flush dos arquivos.

18. Formulário de Projeto (Fase 2)

a) Estrutura usada para representar os registros

Cada registro implementa a interface app.Registro (getId/setId/toByteArray/fromByteArray). A serialização usa DataOutputStream sobre ByteArrayOutputStream, gerando byte[] de tamanho variável. No arquivo binário cada registro é gravado como [lápide:1B][tamanho:short:2B][dados:byte[tamanho]], antecedido pelo cabeçalho global do arquivo (int últimoId + long ponteiroListaRemovidos).

b) Como atributos multivalorados do tipo string foram tratados

Usuario.telefones e Alimento.tags são List<String> e são serializados como [short qtd][writeUTF × qtd]. Na desserialização lemos o short e iteramos chamando readUTF().

c) Como foi implementada a exclusão lógica

Cada registro tem um byte de lápide: ' ' (ativo) ou '*' (removido). O delete sobrescreve o byte com '*' e registra o endereço em uma lista encadeada de espaços livres ordenada por tamanho (best-fit). Em uma inserção/atualização que precise de mais espaço, o método getDeleted(tam) percorre a lista buscando o primeiro nó com tamanho suficiente; caso não exista, o registro é gravado no fim do arquivo.

d) Além das PKs, quais outras chaves foram utilizadas

  • Chave estrangeira (FK) Refeicao.usuarioId — validada na inserção/atualização.
  • FK Consumo.refeicaoId — usada como chave de busca no índice 1:N.
  • FK Consumo.alimentoId — validada para integridade referencial.

e) Como a estrutura (hash) foi implementada para cada chave de pesquisa

A classe HashExtensivel<T extends RegistroHashExtensivel<T>> é totalmente genérica e parametrizada pelo tipo de entrada. Cada índice é instanciado com um número fixo de slots por cesto e um Constructor<T>. O hash usa os profundidadeGlobal bits menos significativos de hashCode() para indexar o diretório. read(chave) retorna todas as entradas do cesto que batem com a chave (suportando 1:N). Quando um cesto enche, ocorre split com possível duplicação do diretório.

ÍndiceTipo de entradachave (hashCode)Slots/cesto
Primário (cada tabela)ParIDEnderecoid8
1:N (Refeicao→Consumo)ParIDIDrefeicaoId16

f) Como foi implementado o relacionamento 1:N

O índice consumo_por_refeicao é um Hash Extensível de ParIDID(FK, PK) que armazena, para cada consumo criado, o par (refeicaoId, consumoId). A operação ConsumoDAO.listarPorRefeicao(refId) consulta o hash com chave refId, obtém todos os consumoIds e em seguida resolve cada um pelo índice primário do arquivo consumo, retornando os objetos Consumo.

A integridade referencial é mantida em três pontos: (1) na criação de um consumo, o controller verifica se a refeição e o alimento existem; (2) ao alterar a refeicaoId de um consumo, o DAO remove o par antigo e insere o novo; (3) ao remover uma refeição, o RefeicaoController consulta o índice 1:N e remove em cascata todos os consumos associados (e suas entradas no índice).

g) Como os índices são persistidos em disco

Cada índice tem dois arquivos em ./dados/{nome}/{nome}.idx.d.db (diretório) e ./dados/{nome}/{nome}.idx.c.db (cestos). O diretório armazena profundidadeGlobal + ponteiros long; cada cesto armazena profundidadeLocal + quantidade + slots de tamanho fixo. As gravações acontecem na mesma operação que altera o arquivo de dados — não há cache em memória — e o servidor fecha todos os RandomAccessFile via shutdown hook. Se o arquivo de índice for apagado, o sistema o reconstrói automaticamente na próxima inicialização varrendo o arquivo .db de dados.
