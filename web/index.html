a

            
Estrutura usada para representar os registros

            
Cada registro implementa a interface app.Registro (getId/setId/toByteArray/fromByteArray). A serialização usa DataOutputStream sobre ByteArrayOutputStream, gerando byte[] de tamanho variável. No arquivo binário cada registro é gravado como:

            
[lápide:1B] [tamanho:short:2B] [dados:byte[tamanho]]

            
O arquivo possui ainda um cabeçalho global de 12 bytes: int últimoId (4B) + long ponteiroListaRemovidos (8B).

          


          

            
b

            
Atributos multivalorados do tipo string

            
Usuario.telefones e Alimento.tags são List<String> e são serializados como:

            
[short qtd] [writeUTF × qtd]

            
Na desserialização lemos o short de quantidade e iteramos chamando readUTF() — formato compacto e auto-delimitado.

          


          

            
c

            
Implementação da exclusão lógica

            
Cada registro tem um byte de lápide: ' ' (ativo) ou '*' (removido). A operação delete sobrescreve o byte com '*' e registra o endereço em uma lista encadeada de espaços livres ordenada por tamanho (estratégia best-fit).

            
Em uma inserção/atualização que precise de mais espaço, o método getDeleted(tam) percorre a lista buscando o primeiro nó com tamanho suficiente; caso não exista, o registro é gravado no fim do arquivo.

          


          

            
d

            
Outras chaves utilizadas além das PKs

            
    
              
  • FK Refeicao.usuarioId — validada na inserção/atualização contra o índice primário de Usuario.
    • 
              
  • FK Consumo.refeicaoId — usada como chave de busca no índice 1:N em Hash Extensível.
    • 
              
  • FK Consumo.alimentoId — validada para integridade referencial.
    • 
            

          


          

            
e

            
Como o Hash Extensível foi implementado para cada chave

            
A classe HashExtensivel<T extends RegistroHashExtensivel<T>> é totalmente genérica e parametrizada pelo tipo de entrada. Cada índice é instanciado com um número fixo de slots por cesto e um Constructor<T>.

            
O hash usa os profundidadeGlobal bits menos significativos de hashCode() para indexar o diretório. read(chave) retorna todas as entradas do cesto que batem com a chave (suportando 1:N). Quando um cesto enche, ocorre split com possível duplicação do diretório.

            

              
ÍndiceTipo de entradaChave (hashCode)Slots/cesto

              
                
Primário (uma por tabela)ParIDEndereco (12B)id8

                
1:N (Refeicao→Consumo)ParIDID (8B)refeicaoId16

              
            

          


          

            
f

            
Como o relacionamento 1:N foi implementado

            
O índice consumo_por_refeicao é um Hash Extensível de ParIDID(FK, PK) que armazena, para cada consumo criado, o par (refeicaoId, consumoId).

            
Navegação: ConsumoDAO.listarPorRefeicao(refId) consulta o hash com chave refId, obtém todos os consumoIds e em seguida resolve cada um pelo índice primário do arquivo consumo, retornando os objetos Consumo.

            
Integridade referencial garantida em três pontos:

            
    
              
  1. Na criação de um consumo, o controller verifica se a refeição e o alimento existem (via exists(id) no índice primário).
    2. 
              
  2. Ao alterar a refeicaoId de um consumo, o DAO remove o par antigo do índice 1:N e insere o novo.
    3. 
              
  3. Ao remover uma refeição, o RefeicaoController consulta o índice 1:N e remove em cascade todos os consumos associados (e suas entradas no índice).
    4. 
            

          


          

            
g

            
Como os índices são persistidos em disco

            
Cada índice tem dois arquivos em ./dados/{nome}/:

            

              
ArquivoConteúdo

              
                
{nome}.idx.d.dbDiretório: int profundidadeGlobal + 2^profGlobal ponteiros long

                
{nome}.idx.c.dbCestos: sequência de cestos com byte profundidadeLocal + short qtd + slots de tamanho fixo

              
            

            
Atualização: as gravações acontecem na mesma operação que altera o arquivo de dados — não há cache em memória — usando RandomAccessFile com seek + write.

            
Sincronização: o servidor fecha todos os arquivos via shutdown hook. Se o arquivo de índice for apagado, o sistema o reconstrói automaticamente na próxima inicialização varrendo o arquivo .db de dados.

1

            
Qual relacionamento N:N foi implementado?

            
O relacionamento "Alimentos Favoritos" entre Usuario e Alimento. Cada usuário pode favoritar muitos alimentos e cada alimento pode ser favoritado por muitos usuários.

            
Foi criada a tabela intermediária Favorito com os atributos: id (PK interna), usuarioId, alimentoId, dataInclusao (data), nota (1–5) e observacao.

          


          

            
2

            
Como a chave primária composta foi modelada?

            
A unicidade do par (usuarioId, alimentoId) é garantida em FavoritoDAO.create: antes de inserir, consulta-se o índice favorito_por_usuario para o usuarioId recebido e verifica-se se algum favorito do usuário aponta para o mesmo alimentoId. Caso exista, a operação é rejeitada com IllegalStateException e o servidor responde 400.

            
Além disso, o FavoritoController.atualizar não permite reatribuir os campos da chave composta — preserva os valores originais do registro armazenado.

          


          

            
3

            
Quais índices auxiliares foram criados para o N:N?

            
Dois índices em Hash Extensível, um para cada direção da consulta:

            

              
ÍndiceEntradaChaveUso

              
                
favorito_por_usuarioParIDID(usuarioId, favoritoId)usuarioIdlistar todos os alimentos favoritados por um usuário

                
favorito_por_alimentoParIDID(alimentoId, favoritoId)alimentoIdlistar todos os usuários que favoritaram um alimento

              
            

            
Ambos são genéricos (HashExtensivel<ParIDID>) e persistidos em ./dados/favorito_por_usuario/ e ./dados/favorito_por_alimento/.

          


          

            
4

            
Como o sistema garante a integridade referencial?

            
    
              
  1. Validação de FKs: FavoritoController.criar verifica que usuarioId e alimentoId existem nos índices primários de Usuário e Alimento antes de inserir.
    2. 
              
  2. Cascade ao remover Usuário: UsuarioController.remover chama favoritoDAO.removerPorUsuario(id), que apaga todos os favoritos vinculados (e suas entradas nos dois índices) antes de remover o usuário.
    3. 
              
  3. Cascade ao remover Alimento: simétrico — AlimentoController.remover chama favoritoDAO.removerPorAlimento(id).
    4. 
              
  4. Unicidade da PK composta: ver questão 2.
    5. 
            

          


          

            
5

            
Qual consulta ordenada foi implementada com Árvore B+?

            
A consulta "listar alimentos em ordem alfabética por nome", exposta em GET /api/alimento/ordenado e demonstrada no card "Listagem ordenada por nome" da tela de Alimentos.

            
É resolvida em duas etapas:

            
    
              
  1. AlimentoDAO.listarOrdenadoPorNome chama idxNome.listarTodos(), que percorre a lista encadeada de folhas da B+ em ordem crescente de chave e devolve uma lista de ParNomeID(nome, id).
    2. 
              
  2. Para cada par, o DAO resolve o registro completo via índice primário (Hash Extensível) com arq.read(id).
    3. 
            

            
Em nenhum momento os registros são trazidos para memória para serem ordenados — a ordem já está materializada nas folhas da árvore.

          


          

            
6

            
Como a Árvore B+ foi implementada?

            
A classe app.dao.ArvoreBMais<T extends RegistroArvoreBMais<T>> é totalmente genérica e persistente. O arquivo de índice tem o layout:

            
offset 0     : long enderecoRaiz     (-1 se vazia)
offset 8+    : páginas de tamanho fixo

[byte tipo]          // 0 = interno, 1 = folha
[short n]            // nº de entradas armazenadas
[long proximo]       // próxima folha (-1 nas internas / última folha)
[entry × ordem]      // slots de entradas (resto zerado)
[long × (ordem+1)]   // ponteiros para filhos (só usados em internas)

            
A ordem usada é 8 e a entrada da B+ de alimento (ParNomeID) ocupa 64 bytes (60 bytes de nome + 4 bytes de id), dando páginas de 595 bytes.

            
Inserção: recursiva, com split de folha (chave do meio é copiada para o pai) e split de interna (chave do meio sobe). Remoção: apaga a entrada na folha sem rebalanceamento. Busca: desce pelos internos e varre folhas enquanto compareTo == 0.

            
O compareTo de ParNomeID normaliza o nome (trim + lowercase) e desempata pelo id, garantindo ordem total e evitando colisões.

          


          

            
7

            
Justificativa: B+ vs Hash Extensível

            
A escolha foi feita por padrão de acesso, não por preferência teórica:

            
    
              
  • Hash Extensível para os dois índices do N:N (favorito_por_usuario, favorito_por_alimento) e para os índices primários (id → endereço). Esses acessos são sempre equality lookup ("dado este id, me dê as associações"), em que o Hash entrega O(1) médio com 1 leitura no diretório + 1 no cesto. Uma B+ daria O(log n) sem ganho real, pois não precisamos de ordenação por essas chaves.
    • 
              
  • Árvore B+ para o índice nome → id do alimento. Aqui o requisito é uma varredura ordenada — a B+ é especialmente boa nisso porque as folhas estão encadeadas em ordem crescente de chave, permitindo recuperar toda a sequência em uma única passada sem ordenar nada em memória. A mesma estrutura ainda permite, no futuro, consultas por faixa (ex.: alimentos entre "a" e "f") com custo proporcional ao tamanho da resposta.
    • 
            

            
Em resumo: usa-se Hash quando o acesso é por igualdade; usa-se B+ quando o acesso é por ordem ou faixa.

Backup & Compressão Fase 4

            
Gera um único arquivo compactado com todos os dados do sistema usando Huffman e LZW, calculando a taxa de compressão.

          

          

            
          

        


        

          
📦 Como funciona

          
    
            
  • Todos os arquivos sob ./dados/ são reunidos em um único pacote (estilo TAR).
    • 
            
  • O pacote é comprimido inteiro por Huffman ou LZW, gerando um único arquivo de backup.
    • 
            
  • Cada geração faz uma verificação de integridade (round-trip): descomprime e compara byte a byte com o original.
    • 
            
  • É possível baixar o arquivo compactado e restaurar em ./dados_restaurado/.
    • 
          

        


        


        

          

            
🌳 Huffman

            
Codificação por prefixos de comprimento variável baseada na frequência dos bytes.

            

              
              
              
            

            

          


          

            
📖 LZW

            
Dicionário adaptativo (Lempel–Ziv–Welch) com códigos de 16 bits.

1

            
Qual foi a taxa de compressão obtida com o algoritmo de Huffman?

            

              
Clique em “Gerar backup Huffman” na aba Backup & Compressão para calcular automaticamente os valores abaixo.

            

            
c) Cálculo da taxa: taxa = (1 − tamanhoComprimido / tamanhoOriginal) × 100

            
d) Interpretação: O Huffman explora a frequência dos bytes — quanto mais repetitivos/enviesados os dados, menor a entropia e melhor a taxa. Como os arquivos .db contêm muitos bytes de preenchimento (lápides, espaços livres e campos fixos), há boa redundância estatística a explorar. Por ser um código de prefixo ótimo para símbolos isolados, ele atinge a entropia de ordem 0, mas não captura padrões de sequências repetidas.

          


          

            
2

            
Qual foi a taxa de compressão obtida com o algoritmo de LZW?

            

              
Clique em “Gerar backup LZW” na aba Backup & Compressão para calcular automaticamente os valores abaixo.

            

            
c) Cálculo da taxa: taxa = (1 − tamanhoComprimido / tamanhoOriginal) × 100

            
d) Interpretação: O LZW monta um dicionário de sequências recorrentes. Nos arquivos de dados há padrões repetidos (cabeçalhos, sequências de bytes nulos, strings comuns), que o LZW substitui por um único código. Por isso, em arquivos estruturados como os .db, ele tende a alcançar taxa igual ou superior à do Huffman, capturando redundância de sequência que o Huffman puro não enxerga.

          


          

            
3

            
Quais dificuldades surgiram ao implementar Huffman e LZW e como você resolveu?

            
    
              
  • Empacotar tudo em um arquivo único: os dados estão espalhados em várias pastas/arquivos. Resolvido com um empacotador estilo TAR (Backup.empacotar) que serializa caminho relativo + tamanho + conteúdo de cada arquivo num só fluxo antes de comprimir.
    • 
              
  • Huffman — persistir a árvore: para descomprimir é preciso reconstruir exatamente a mesma árvore. Resolvido gravando a tabela de frequências no cabeçalho do fluxo e reconstruindo a árvore de forma determinística (fila de prioridade).
    • 
              
  • Huffman — empacotamento de bits: os códigos têm tamanho variável e não se alinham a bytes. Resolvido com um bit buffer que acumula bits e descarrega de 8 em 8, preenchendo o último byte com zeros.
    • 
              
  • Huffman — caso de símbolo único: um arquivo com um só byte distinto gera árvore de um nó. Resolvido com tratamento especial, atribuindo o código “0” e detectando a raiz-folha na descompressão.
    • 
              
  • LZW — sincronizar o dicionário: compressor e descompressor precisam construir dicionários idênticos. Resolvido parando o crescimento exatamente no mesmo limite (65.536 entradas) e tratando o caso especial KwKwK (código ainda não presente no dicionário do leitor).
    • 
              
  • Integridade: garantir que nada se perde. Resolvido com verificação round-trip automática (comprime → descomprime → compara byte a byte) antes de gravar o arquivo de backup.
    •
