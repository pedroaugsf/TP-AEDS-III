a

            
Estrutura usada para representar os registros

            
Cada registro implementa a interface app.Registro (getId/setId/toByteArray/fromByteArray). A serialização usa DataOutputStream sobre ByteArrayOutputStream, gerando byte[] de tamanho variável. No arquivo binário cada registro é gravado como:

            
[lápide:1B] [tamanho:short:2B] [dados:byte[tamanho]]

            
O arquivo possui ainda um cabeçalho global de 12 bytes: int últimoId (4B) + long ponteiroListaRemovidos (8B).

          


          

            
b

            
Atributos multivalorados do tipo string

            
Usuario.telefones e Alimento.tags são List<String> e são serializados como:

            
[short qtd] [writeUTF × qtd]

            
Na desserialização lemos o short de quantidade e iteramos chamando readUTF() — formato compacto e auto-delimitado.

          


          

            
c

            
Implementação da exclusão lógica

            
Cada registro tem um byte de lápide: ' ' (ativo) ou '*' (removido). A operação delete sobrescreve o byte com '*' e registra o endereço em uma lista encadeada de espaços livres ordenada por tamanho (estratégia best-fit).

            
Em uma inserção/atualização que precise de mais espaço, o método getDeleted(tam) percorre a lista buscando o primeiro nó com tamanho suficiente; caso não exista, o registro é gravado no fim do arquivo.

e

            
Como o Hash Extensível foi implementado para cada chave

            
A classe HashExtensivel<T extends RegistroHashExtensivel<T>> é totalmente genérica e parametrizada pelo tipo de entrada. Cada índice é instanciado com um número fixo de slots por cesto e um Constructor<T>.

            
O hash usa os profundidadeGlobal bits menos significativos de hashCode() para indexar o diretório. read(chave) retorna todas as entradas do cesto que batem com a chave (suportando 1:N). Quando um cesto enche, ocorre split com possível duplicação do diretório.

            

              
ÍndiceTipo de entradaChave (hashCode)Slots/cesto

              
                
Primário (uma por tabela)ParIDEndereco (12B)id8

                
1:N (Refeicao→Consumo)ParIDID (8B)refeicaoId16

              
            

          


          

            
f

            
Como o relacionamento 1:N foi implementado

            
O índice consumo_por_refeicao é um Hash Extensível de ParIDID(FK, PK) que armazena, para cada consumo criado, o par (refeicaoId, consumoId).

            
Navegação: ConsumoDAO.listarPorRefeicao(refId) consulta o hash com chave refId, obtém todos os consumoIds e em seguida resolve cada um pelo índice primário do arquivo consumo, retornando os objetos Consumo.

            
Integridade referencial garantida em três pontos:

            
    
              
  1. Na criação de um consumo, o controller verifica se a refeição e o alimento existem (via exists(id) no índice primário).
    2. 
              
  2. Ao alterar a refeicaoId de um consumo, o DAO remove o par antigo do índice 1:N e insere o novo.
    3. 
              
  3. Ao remover uma refeição, o RefeicaoController consulta o índice 1:N e remove em cascade todos os consumos associados (e suas entradas no índice).
    4. 
            

          


          

            
g

            
Como os índices são persistidos em disco

            
Cada índice tem dois arquivos em ./dados/{nome}/:

            

              
ArquivoConteúdo

              
                
{nome}.idx.d.dbDiretório: int profundidadeGlobal + 2^profGlobal ponteiros long

                
{nome}.idx.c.dbCestos: sequência de cestos com byte profundidadeLocal + short qtd + slots de tamanho fixo

              
            

            
Atualização: as gravações acontecem na mesma operação que altera o arquivo de dados — não há cache em memória — usando RandomAccessFile com seek + write.

            
Sincronização: o servidor fecha todos os arquivos via shutdown hook. Se o arquivo de índice for apagado, o sistema o reconstrói automaticamente na próxima inicialização varrendo o arquivo .db de dados.
